# 03. Business Process

## 1. 문서 목적

본 문서는 업무관리시스템(WMS)의 업무 프로세스, 권한 정책, 상태 정책을 정의한다.

본 문서에서 정의한 내용은 이후 ERD 설계, 화면 설계, API 설계 및 개발의 기준으로 사용한다.

---

# 2. 공통 정책

## 2.1 승인 / 반려 정책

모든 승인 및 반려 행위는 이력으로 관리한다.

정책

* 승인 및 반려 이력은 삭제할 수 없다.
* 반려 사유 입력은 필수이다.
* 반려 시 항상 직전 단계로 복귀한다.
* 모든 상태 변경은 이력으로 저장한다.

---

## 2.2 권한(Role)과 업무 주체(Actor)

시스템 권한(Role)과 업무 주체(Actor)는 별도로 관리한다.

### Role

* USER
* SR_MANAGER
* DEPT_MANAGER
* DEV_MANAGER
* HR_MANAGER
* FACILITY_MANAGER
* ADMIN

### Actor

* SR 요청자
* SR 담당자
* SR 개발팀장
* 전자결재 기안자
* 전자결재 승인자
* 전자결재 참조자
* 전자결재 대직자
* 직원등록 요청자
* 직원등록 승인자

---

## 2.3 권한 중복 보유 정책

사용자는 여러 권한을 동시에 보유할 수 있다.

예시

* USER + SR_MANAGER
* USER + DEPT_MANAGER
* USER + DEV_MANAGER

---

## 2.4 권한 우선순위 정책

동일 화면 및 기능에서 권한 충돌 발생 시 우선순위가 높은 권한을 적용한다.

| 권한               | 우선순위 |
| ---------------- | ---: |
| ADMIN            |  100 |
| DEV_MANAGER      |   90 |
| HR_MANAGER       |   80 |
| FACILITY_MANAGER |   70 |
| DEPT_MANAGER     |   60 |
| SR_MANAGER       |   50 |
| USER             |   10 |

---

## 2.5 코드 관리 정책

변경 가능성이 있는 업무 데이터는 코드 관리 대상으로 정의한다.

예시

* 직급
* 전자결재 종류
* 시설 종류
* 공지 범위
* SR 우선순위

변경 가능성이 없는 시스템 내부 상태값은 Enum으로 관리한다.

예시

* SR 상태값
* 버튼 정책
* 액션 정책
* 알림 유형

---

## 2.6 대직 정책

사용자는 대직자를 지정할 수 있다.

대직 정보

* 대직자
* 시작일
* 종료일
* 대직 사유

대직 기간 동안 승인 및 반려 처리가 가능하다.

대직 기간 종료 시 대직 권한은 자동 종료된다.

---

## 2.7 첨부파일 정책

첨부파일은 공통 정책을 사용한다.

지원 확장자

* pdf
* doc
* docx
* xls
* xlsx
* ppt
* pptx
* jpg
* jpeg
* png
* zip

제한 확장자

* exe
* bat
* msi

---

## 2.8 알림 정책

시스템 내부 알림을 제공한다.

알림 발생 예시

* SR 승인 요청
* SR 담당자 지정
* SR 인수 테스트 요청
* 전자결재 승인 요청
* 직원 등록 승인 요청

알림 정보

* 제목
* 내용
* 링크
* 읽음 여부
* 읽음 일시

---

## 2.9 알림 배치 정책

정기 배치를 통해 미처리 업무를 확인한다.

대상

* SR
* 전자결재
* 직원 등록

미처리 상태가 일정 기간 지속될 경우 자동 알림을 생성한다.

---

# 3. 권한(Role) 정의

## USER

기본 권한

가능 기능

* 대시보드 조회
* 내 정보 조회
* 내 정보 수정
* SR 요청
* SR 현황 조회
* 전자결재 작성
* 결재함 조회
* 시설 예약
* 직원 조회
* 조직도 조회

---

## SR_MANAGER

개발팀 직원 기본 권한

SR_MANAGER 권한은 개발팀 소속 직원에게 자동 부여한다.

가능 기능

* SR 진행 메뉴 접근
* SR 담당자 가져오기
* SR 담당자 이첩
* SR 의견 등록

---

## DEPT_MANAGER

부서 팀장 권한

가능 기능

* SR 1차 승인
* SR 1차 반려
* 직원 등록 요청
* 부서 공지사항 등록

---

## DEV_MANAGER

개발팀장 권한

가능 기능

* SR 최종 승인
* SR 최종 반려
* 담당자 지정
* 담당자 변경
* SR 우선순위 확정
* SR 종결 승인
* 시스템 환경설정 관리

---

## HR_MANAGER

인사 담당 권한

가능 기능

* 직원 등록 승인
* 직원 등록 반려
* 직원 정보 관리
* 부서 이동
* 권한 관리
* 전체 공지사항 등록

---

## FACILITY_MANAGER

시설 담당 권한

가능 기능

* 시설 등록
* 시설 수정
* 시설 비활성화

---

## ADMIN

최초 시스템 구축용 관리자

운영 환경에서는 사용을 최소화한다.

---

# 4. SR 프로세스

## 4.1 상태 정의

SR 상태는 Enum으로 관리한다.

| 상태                 |
| ------------------ |
| REQUESTED          |
| WAIT_DEPT_APPROVAL |
| WAIT_DEV_APPROVAL  |
| ASSIGNED           |
| IN_PROGRESS        |
| WAIT_UAT           |
| WAIT_CLOSE         |
| COMPLETED          |

---

## 4.2 상태 관리 정책

SR 상태는 코드 관리 대상이 아니다.

상태별 버튼 노출 및 액션 가능 여부는 다음 기준으로 판단한다.

* 상태(State)
* 권한(Role)
* 사용자(User)

---

## 4.3 처리 흐름

요청자

↓

부서 팀장 승인

↓

개발팀장 승인

↓

담당자 지정

↓

개발 진행

↓

인수 테스트 요청

↓

요청자 인수 테스트

↓

개발팀장 종결 승인

↓

종결

---

## 4.4 반려 정책

부서 팀장 반려

↓

요청자

---

개발팀장 반려

↓

부서 팀장 승인 단계

---

인수 테스트 반려

↓

개발 진행 단계

---

개발팀장 종결 반려

↓

인수 테스트 단계

---

## 4.5 우선순위 정책

요청자 선택

* 긴급
* 일반

개발팀장이 최종 확정한다.

---

## 4.6 담당자 정책

개발팀장은 담당자를 지정할 수 있다.

개발팀 직원은

* 담당자 가져오기
* 담당자 이첩

처리가 가능하다.

SR 상태 변경은 현재 담당자만 가능하다.

---

## 4.7 업무 주체

### 요청자

가능 기능

* 등록
* 수정
* 취소
* 현황 조회

### 개발팀장

가능 기능

* 승인
* 반려
* 담당자 지정
* 담당자 변경

### 개발팀 직원

가능 기능

* 담당자 가져오기
* 담당자 이첩

### SR 담당자

가능 기능

* 진행 처리
* 의견 등록
* 완료 요청

---

## 4.8 상태 변경 권한

| 상태                 | 처리자          | 가능 액션   |
| ------------------ | ------------ | ------- |
| WAIT_DEPT_APPROVAL | DEPT_MANAGER | 승인 / 반려 |
| WAIT_DEV_APPROVAL  | DEV_MANAGER  | 승인 / 반려 |
| ASSIGNED           | 담당자          | 진행 시작   |
| IN_PROGRESS        | 담당자          | 완료 요청   |
| WAIT_UAT           | 요청자          | 승인 / 반려 |
| WAIT_CLOSE         | DEV_MANAGER  | 종결 / 반려 |
